Heroku is my Hero

Published: Wed 30 July 2014
By Terrel Shumway

In programming.

tags: django web heroku postgresql

While working on a project for school, I was looking for a simple way to deploy a Django site. I discovered Heroku. I had heard about Heroku a while back, but got the impression that it was mostly for Ruby, so I didn't investigate too much. (Ruby is a fine language, but for me it doesn't provide enough marginal benefit over Python to justify the effort to learn it well. Maybe someday, but not now.) This latest endorsement of Heroku was by Jacob Kaplan-Moss, one of the co-founders of the Django project, so I took it seriously.

I was quite pleasantly surprised. I am now officially spoiled. Deploying an app was about as simple as it could possibly be.

It also highlighted a huge hole in my learning. Heroku uses PostgresQL. Sqlite need not apply because the heroku filesystem is ephemeral. You can create a sqlite database, but as soon as the dyno goes down, your data is gone too. In all of my Djangoing I had never used anything but sqlite. Oops. Then I read about how developing on sqlite (even testing) is a puerile newbie mistake. I guess I really am a newbie.

Fine. If Heroku can use PostgresQL, so can I. Then I discover that I really don't want to open that can of worms just yet. I'll use sqlite on my development machine, and PostgresQL on heroku. For now. I am beginning to understand why DBAs get paid the big bucks.
